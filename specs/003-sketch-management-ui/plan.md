# Implementation Plan: Sketch Management UI

**Branch**: `003-sketch-management-ui` | **Date**: 2026-06-23 | **Spec**: [spec.md](./spec.md)

**Input**: Feature specification from `specs/003-sketch-management-ui/spec.md`

## Summary

Add a fully in-browser sketch management experience — create, duplicate, edit (rename/retype/re-slug), and delete — backed by four Vite dev-server middleware endpoints that reuse the existing `scripts/lib/` utilities. All mutations are local-only (gated to `apply: 'serve'`), require no auth, and rely on Vite HMR to refresh the gallery automatically. A new `scripts/edit-sketch.ts` CLI script + shared `scripts/lib/edit-sketch-op.ts` module support slug/folder renaming. The gallery gains an Actions column (duplicate, edit, delete icon buttons) and a New Sketch header button; both the gallery and the individual sketch page expose edit and delete. All forms are Radix Dialog modals sharing a single `SketchFormDialog` component.

## Technical Context

**Language/Version**: TypeScript 5.7, Node.js 22+ (server middleware), React 19 (browser UI)

**Primary Dependencies**: Existing — Vite 6, React 19, React Router 7, Tailwind v4, TanStack Table, zod, `scripts/lib/*`. New — `@radix-ui/react-dialog` (modal primitives for forms + confirmation prompts)

**Storage**: Filesystem only — `sketches/<id>/meta.json` + `sketch.ts` (+ optional `globals.ts`)

**Testing**: Vitest (unit: middleware logic, slug derivation, edit-sketch-op); React Testing Library (component: dialog open/close, form validation, actions column rendering)

**Target Platform**: Local developer machine, modern evergreen browser, Vite dev server (`npm run dev` only)

**Project Type**: Web application (frontend SPA) + Node.js dev-server plugin + CLI tooling, single npm package

**Performance Goals**: Each mutation (create/duplicate/edit/delete) completes within 3 s including HMR-triggered page reload (SC-005). Form validation feedback within 1 s of invalid submission (SC-004).

**Constraints**: Mutations MUST be unavailable in static production builds (FR-019). `update-sketch-meta` fields (`dateUpdated`, `lastUpdatedBy`) are never written by UI-triggered mutations (FR-022). Slug/id field is absent from the sketch-page edit form (FR-012).

## Implementation Phases

### Phase A — Server layer (no UI changes)

**Goal**: All four API endpoints work and are verifiable with `curl`.

1. **`scripts/lib/edit-sketch-op.ts`** — pure function `editSketch({ id, name, newId?, type })`:
   - Validate inputs (non-empty name, valid type, valid newId if provided)
   - Check source sketch exists
   - If `newId` provided and `newId !== id`: check target does not exist, `fs.renameSync`, update `id` in meta
   - `writeMeta(finalId, { ...existingMeta, name, type })` (preserves `dateCreated`, `createdBy`, `dateUpdated`, `lastUpdatedBy`)

2. **`scripts/edit-sketch.ts`** — CLI wrapper: `npm run edit-sketch -- --id <id> [--name <n>] [--new-id <nid>] [--type <t>]`

3. **`src/plugins/sketch-api.ts`** — Vite plugin (`apply: 'serve'`, `configureServer`):
   - Parse JSON body manually (or use a tiny utility — no Express needed in Vite middleware)
   - `POST /api/sketches` → delegates to create-sketch logic from `scripts/lib/`
   - `POST /api/sketches/:sourceId/duplicate` → delegates to duplicate-sketch logic
   - `PATCH /api/sketches/:id` → delegates to `editSketch()`
   - `DELETE /api/sketches/:id` → delegates to delete-sketch logic
   - All return `{ ok: true }` or `{ ok: false, error }` JSON

4. **`vite.config.ts`** — import and add `sketchApi()` to the `plugins` array

### Phase B — Shared form component

**Goal**: A single dialog handles create, duplicate, and edit with correct field visibility.

5. **Install `@radix-ui/react-dialog`**

6. **`src/components/SketchFormDialog.tsx`** — props: `mode: 'create' | 'duplicate' | 'edit-full' | 'edit-name-only'`, `initialValues?`, `onSubmit(values)`, `onClose()`:
   - Name field: always present
   - Slug/id field: present in `create`, `duplicate`, `edit-full`; absent in `edit-name-only`
   - Auto-derivation: `slugify(name)` on every name keystroke until user manually edits slug (sets `slugDirty`)
   - Type selector: present in all modes
   - Client-side validation: non-empty name, valid slug pattern, non-empty type
   - Server-side conflict error: displayed inline after API call
   - Submit/cancel buttons; Escape dismisses

7. **`src/components/DeleteConfirmDialog.tsx`** — props: `sketchName`, `onConfirm()`, `onClose()`:
   - One-sentence confirmation, sketch name displayed, Delete + Cancel buttons

### Phase C — Gallery integration

**Goal**: Actions column in the gallery table; New Sketch button in the page header.

8. **`src/components/SketchTable.tsx`** — add `actions` display column (rightmost):
   - Each row: duplicate icon button, edit icon button, delete icon button
   - Clicking a row body still navigates; icon buttons use `e.stopPropagation()`
   - Icon buttons use existing `Button` component or small `<button>` styled with Tailwind

9. **`src/pages/HomePage.tsx`** — add "New Sketch" button above or alongside the gallery header; manages `dialogOpen` state for `SketchFormDialog` (mode: `create`)

10. **Wire gallery mutations**: each dialog/confirm `onSubmit` calls the relevant API endpoint, awaits the response, handles errors inline; on success closes the dialog (HMR/reload follows naturally)

### Phase D — Sketch page integration

**Goal**: Edit and Delete accessible from the sketch page; edit does not expose slug.

11. **`src/pages/SketchPage.tsx`** — add Edit and Delete buttons to the top-right action area (alongside the existing "← All sketches" link):
    - Edit → opens `SketchFormDialog` in `edit-name-only` mode with current `meta.name` + `meta.type`
    - Delete → opens `DeleteConfirmDialog`; on confirm calls `DELETE /api/sketches/:id`, then navigates to `/`

### Phase E — Tests

12. **Middleware unit tests** (`tests/unit/sketch-api.test.ts`): test each endpoint handler's logic against a temp directory — create, duplicate, edit (name-only, with slug rename), delete; slug conflict; source-not-found errors.

13. **`edit-sketch-op` unit tests** (`tests/unit/edit-sketch-op.test.ts`): covers rename path, name-only path, type change, conflict rejection.

14. **Component tests** (`tests/components/SketchFormDialog.test.tsx`, `DeleteConfirmDialog.test.tsx`): slug auto-derivation, slugDirty flag, validation errors, mode-dependent field visibility (no slug in `edit-name-only`).

15. **SketchTable actions column test** (`tests/components/SketchTable.test.tsx`): actions column renders three icon buttons per row; button click does not trigger row navigation.

## Key Design Decisions

See `research.md` for full rationale on each. Summary:

| Decision | Choice |
|----------|--------|
| Server mechanism | Vite `configureServer` plugin, `apply: 'serve'` |
| Gallery refresh | Vite HMR file watcher — no explicit reload call needed |
| Slug in browser | Import `slugify`/`isValidSlug` from `scripts/lib/slug.ts` directly |
| Edit atomicity | Conflict-check → `renameSync` → `writeMeta` (local filesystem, acceptable risk) |
| Modal library | `@radix-ui/react-dialog` (extends existing Radix usage) |
| Edit CLI + middleware | Shared `scripts/lib/edit-sketch-op.ts` module |
| API response | `{ ok: boolean; error?: string }`, HTTP 200/400/500 |
| CI fields | `dateUpdated` / `lastUpdatedBy` never written by UI mutations |
